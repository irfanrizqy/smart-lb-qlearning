# PATCH-2026-05-17

## Patch Perbaikan Q-Learning Load Balancer, Training Gate, Runtime Monitoring, dan Redis Inspector

Tanggal: 17 Mei 2026  
Project: Smart Load Balancer berbasis Q-Learning  
Fokus patch: memperbaiki pemisahan antara **routing/inference runtime** dan **training/learning process**, sekaligus membuat Redis monitoring lebih jelas dan tidak misleading.

---

## 1. Ringkasan Masalah Awal

Sebelum patch ini, sistem masih berjalan tetapi ada beberapa masalah konseptual dan teknis yang bisa merusak validitas eksperimen:

1. `qlearning_cycle` terus bertambah meskipun training gate disabled. Ini sebenarnya wajar karena loop routing tetap berjalan, tetapi belum ada counter terpisah untuk cycle yang benar-benar melakukan learning.
2. Saat training OFF, sistem masih bisa memilih action dengan mode `EXPLORE` karena action dipilih menggunakan epsilon-greedy sebelum training gate dicek.
3. Monitoring Redis membaca `qlearning_stats`, padahal saat training OFF key tersebut bisa stale atau berisi data lama dari training sebelumnya.
4. `redis_inspect.sh` belum menampilkan perbedaan antara runtime cycle dan training cycle.
5. `redis_inspect.sh` belum menampilkan `qlearning_routing_log`, padahal routing log tetap penting saat training OFF.
6. Q-table inspector masih berpotensi hardcode jumlah state lama, sementara state Q-Learning sudah berubah karena response time masuk sebagai dimensi state.
7. Saat training flag diubah ON/OFF di tengah satu cycle, action mode dan runtime status bisa terlihat “kebalik” karena training flag dibaca lebih dari sekali dalam satu cycle.

Patch ini memperbaiki masalah-masalah tersebut agar eksperimen load balancing lebih valid secara teknis dan metodologis.

---

## 2. Prinsip Desain Setelah Patch

Setelah patch ini, sistem dibagi jelas menjadi dua jenis cycle:

```text
qlearning_cycle          = runtime/routing cycle
qlearning_training_cycle = cycle yang benar-benar lolos training gate dan melakukan learning
```

Dengan prinsip behavior berikut:

```text
Training OFF:
- routing tetap berjalan
- state tetap diobservasi
- action tetap dipilih
- current_weights tetap ditulis
- selected_action tetap ditulis
- heartbeat tetap ditulis
- routing log tetap bertambah
- reward tidak dihitung untuk learning
- Q-table tidak update
- qlearning_history tidak bertambah
- epsilon tidak decay
- action mode harus EVAL, bukan EXPLORE
```

```text
Training ON + traffic valid:
- routing berjalan
- reward dihitung
- Q-table update
- qlearning_history bertambah
- epsilon decay
- qlearning_training_cycle bertambah
```

```text
Training ON + traffic tidak valid / idle:
- routing tetap berjalan
- action boleh epsilon-greedy karena training mode sedang aktif
- Q-table tetap tidak update karena throughput belum valid
- qlearning_training_cycle tidak bertambah
- runtime status menjadi IDLE_NO_TRAFFIC
```

## 3. Perubahan pada `loop.py`

### 3.1 Menambahkan `qlearning_training_cycle`

#### Sebelum

Hanya ada runtime cycle:

```python
cycle = int(redis_client.get("qlearning_cycle") or 0)
```

#### Sesudah

Ditambahkan counter khusus training:

```python
cycle = int(redis_client.get("qlearning_cycle") or 0)
training_cycle = int(redis_client.get("qlearning_training_cycle") or 0)
```

#### Tujuan

Membedakan antara:

- cycle runtime yang selalu berjalan untuk routing;
- cycle training yang hanya bertambah saat sistem benar-benar melakukan learning.

Ini penting karena `qlearning_cycle` memang tetap bertambah walaupun training OFF. Yang tidak boleh bertambah saat training OFF adalah `qlearning_training_cycle`.

---

### 3.2 Runtime cycle tetap bertambah di awal loop

#### Tetap dipertahankan

```python
cycle += 1
redis_client.set("qlearning_cycle", cycle)
write_heartbeat()
```

#### Tujuan

`qlearning_cycle` tetap merepresentasikan aktivitas runtime/routing. Ini tidak dihentikan saat training OFF karena load balancer tetap harus mengambil keputusan routing dan menjaga heartbeat agar xDS tidak fallback.

---

### 3.3 Snapshot training flag di awal cycle

#### Perubahan

Pada STEP 2, training flag dibaca sekali dan disimpan sebagai snapshot:

```python
training_enabled_for_cycle = is_training_enabled()

if training_enabled_for_cycle:
    action, mode = select_action(q_table, state, epsilon)
else:
    action, _ = select_action(q_table, state, 0.0)
    mode = "EVAL"
```

#### Tujuan

Menghindari kondisi race ketika `qlearning_training_enabled` berubah di tengah cycle.

Sebelumnya, training flag bisa terbaca ON saat action dipilih, lalu terbaca OFF saat training gate dicek setelah `wait_with_heartbeat()`. Akibatnya monitoring terlihat aneh, misalnya:

```text
Training: DISABLED
Runtime : TRAINING_DISABLED
Last    : Mode: EXPLORE
```

Dengan snapshot ini, satu cycle memakai status training yang konsisten dari awal sampai akhir.

---

### 3.4 Training OFF dipaksa memakai mode `EVAL`

#### Perubahan

Saat training OFF, action tidak lagi dipilih memakai epsilon aktif:

```python
action, _ = select_action(q_table, state, 0.0)
mode = "EVAL"
```

#### Tujuan

Saat training OFF, sistem tidak boleh exploration/random. Training OFF dipakai untuk:

- capacity validation;
- learned-policy evaluation;
- komparasi final Q-Learning vs WRR;
- menjaga Q-table freeze.

Jika saat training OFF sistem masih `EXPLORE`, hasil evaluasi bisa kotor karena routing masih dipengaruhi random action, bukan policy hasil Q-value.

## 7. Perubahan pada `config.py`

### 7.1 Response time threshold disarankan disesuaikan dengan capacity pilot

#### Sebelum

```python
RT_LEVEL_THRESHOLDS = [100, 300, 600]
NUM_RT_LEVELS       = 4
TOTAL_STATE_SPACE = (NUM_LEVELS ** NUM_BACKENDS) * NUM_RT_LEVELS  # 5^3 × 4 = 500
```

#### Rekomendasi setelah capacity pilot

```python
RT_LEVEL_THRESHOLDS = [50, 75, 100, 150]
NUM_RT_LEVELS       = 5

NUM_BACKENDS      = len(BACKENDS)
TOTAL_STATE_SPACE = (NUM_LEVELS ** NUM_BACKENDS) * NUM_RT_LEVELS  # 5^3 × 5 = 625
```

#### Tujuan

Karena hasil capacity pilot menunjukkan response time normal banyak berada di bawah 100 ms, threshold lama terlalu longgar. Dengan threshold baru, Q-Learning lebih sensitif terhadap degradasi response time.

Catatan: jika threshold ini belum diterapkan, monitor masih akan menunjukkan total state `500`. Jika sudah diterapkan dan service sudah restart, total state akan menjadi `625`.

### 8.1 Logic `get_rt_level()` tetap aman

Fungsi ini tidak wajib diubah karena sudah fleksibel:

```python
def get_rt_level(rt_ms):
    for i, threshold in enumerate(RT_LEVEL_THRESHOLDS):
        if rt_ms < threshold:
            return i
    return len(RT_LEVEL_THRESHOLDS)
```

Jika `RT_LEVEL_THRESHOLDS` berisi 4 threshold, output otomatis menjadi 5 level.

---

### 8.2 Update komentar/docstring RT level

#### Komentar baru yang disarankan

```python
# [ST-3] RT masuk state sebagai dimensi ke-4
# 5 level: FAST/NORMAL/ELEVATED/SLOW/CRIT.
```

#### Docstring baru yang disarankan

```text
L0: < 50ms    (FAST)
L1: 50–75ms   (NORMAL)
L2: 75–100ms  (ELEVATED)
L3: 100–150ms (SLOW)
L4: >= 150ms  (CRIT)
```

#### Tujuan

Menyelaraskan dokumentasi source code dengan state space baru.

## 13. Kesimpulan Patch

Patch ini menyelesaikan inti masalah desain Q-Learning load balancer:

1. Routing runtime dan training learning sudah dipisahkan.
2. Training OFF tidak lagi berarti service berhenti; routing tetap aktif.
3. Training OFF sekarang memakai mode EVAL, bukan EXPLORE.
4. Runtime cycle dan training cycle sudah dibedakan.
5. Redis monitoring tidak lagi misleading karena membaca `qlearning_runtime` saat training OFF.
6. Routing log bisa diperiksa langsung dari Redis Inspector.
7. Reset Redis lebih bersih karena mencakup key baru.
8. Q-table inspector siap untuk state baru yang menyertakan response time.
9. Race condition akibat toggle training ON/OFF di tengah cycle dikurangi dengan snapshot training flag per cycle.

Dengan patch ini, sistem lebih layak untuk digunakan pada eksperimen akhir:

```text
capacity validation → training Q-Learning → freeze policy → evaluasi WRR vs Q-Learning
```
