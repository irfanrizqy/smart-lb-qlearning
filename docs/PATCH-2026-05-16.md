## Ringkasan

Patch ini menambahkan dua fitur adaptif utama pada Q-Learning engine (RT dalam STATE dan Adaptive Epsilon), memperbaiki sejumlah bug kritis, menghapus dead code, dan mengoptimalkan performa query serta robustness server.

## 1. Fitur Baru

### [ST-3] Response Time sebagai Dimensi ke-4 State
**File:** `qlearning/state.py`, `config.py`

State Q-Learning diperluas dari 3-tuple `(vm3, vm4, vm5)` menjadi 4-tuple `(vm3, vm4, vm5, rt_level)`.

- Fungsi baru `get_rt_level(rt_ms)` memetakan response time (ms) ke 4 level diskrit:
  - L0: < 100ms — FAST
  - L1: 100–300ms — NORMAL
  - L2: 300–600ms — SLOW
  - L3: ≥ 600ms — CRIT
- `observe_state(last_rt_ms=0.0)` menerima RT cycle sebelumnya karena nilai baru baru tersedia setelah STEP 7 (setelah wait).
- State space naik dari **125 (5³)** menjadi **500 (5³ × 4)**, Q-table dari 375 sel menjadi 1500 sel.

**Konstanta baru di `config.py` `[CFG-8]`:**
```python
RT_LEVEL_THRESHOLDS = [100, 300, 600]   # ambang level RT (ms)
NUM_RT_LEVELS       = 4
TOTAL_STATE_SPACE   = (NUM_LEVELS ** NUM_BACKENDS) * NUM_RT_LEVELS  # 500
```

---

### [LP-5] Integrasi RT dalam Loop
**File:** `qlearning/loop.py`

- Variabel `last_rt_ms = 0.0` dilacak antar cycle.
- Kedua `observe_state()` (STEP 1 dan STEP 5) dipanggil dengan `last_rt_ms`.
- Setelah STEP 7, `last_rt_ms` diperbarui jika `response_time` valid.
- Semua log state menampilkan dimensi ke-4 (FAST/NORMAL/SLOW/CRIT) via konstanta `RT_LEVEL_NAMES`.

---

### [LP-6] Adaptive Epsilon
**File:** `qlearning/loop.py`, `config.py`

Epsilon di-boost otomatis saat performa Q-Learning menurun. Mekanisme:
- Setiap cycle (setelah `reward_history` punya ≥ 40 entry), bandingkan avg reward 20 cycle terbaru vs 20 sebelumnya.
- Jika selisih < `ADAPTIVE_EPSILON_DEGRADATION` (−0.15), epsilon di-boost sebesar `ADAPTIVE_EPSILON_BOOST` (0.10).
- Epsilon tidak melebihi `ADAPTIVE_EPSILON_MAX_BOOST` (0.30).

**Konstanta baru di `config.py` `[CFG-9]`:**
```python
ADAPTIVE_EPSILON_WINDOW      = 20
ADAPTIVE_EPSILON_DEGRADATION = -0.15
ADAPTIVE_EPSILON_BOOST       = 0.10
ADAPTIVE_EPSILON_MAX_BOOST   = 0.30
```

---

### [LP-7] Cooldown Adaptive Epsilon
**File:** `qlearning/loop.py`, `qlearning/runtime.py`, `config.py`

Mencegah epsilon di-boost setiap cycle selama degradasi berkepanjangan.

- Setelah boost, blokir boost berikutnya selama `ADAPTIVE_EPSILON_COOLDOWN` cycle.
- Cooldown dikurangi 1 per cycle (baik saat degradasi maupun normal), sehingga recovery alami terjadi lebih cepat jika performa membaik.
- Cooldown dipersist ke Redis via `save_runtime_state()` sehingga tidak hilang saat service restart.

**Konstanta baru di `config.py` `[CFG-10]`:**
```python
ADAPTIVE_EPSILON_COOLDOWN = 10
```

---

### [CFG-11] Action Bias Dapat Dikonfigurasi
**File:** `config.py`, `qlearning/weights.py`

`ACTION_BIAS_EXPLORE` dan `ACTION_BIAS_EXPLOIT` dipindahkan dari `weights.py` ke `config.py` agar dapat dikonfigurasi tanpa edit source file.

```python
ACTION_BIAS_EXPLORE = 15.0   # bias saat EXPLORE (lebih besar — action random diuji penuh)
ACTION_BIAS_EXPLOIT = 10.0   # bias saat EXPLOIT (lebih kecil — Q-values sudah memandu)
```

## 2. Bug Fix

### [QT-1] `state_key()` Membuang Dimensi ke-4 (Bug Kritis)
**File:** `qlearning/qtable.py`

**Sebelum:**
```python
def state_key(state):
    return f"{state[0]}_{state[1]}_{state[2]}"  # hardcode 3 elemen
```

**Sesudah:**
```python
def state_key(state):
    return "_".join(str(s) for s in state)  # dynamic, semua elemen
```

Tanpa fix ini, state `(1,2,0,0)` (RT FAST) dan `(1,2,0,3)` (RT CRIT) dipetakan ke entry Q-table yang sama, membuat fitur RT dalam STATE sepenuhnya tidak bekerja.

---

### [RT-1] `save_runtime_state()` Tidak Persist Cooldown
**File:** `qlearning/runtime.py`

Parameter `adaptive_epsilon_cooldown` ditambahkan ke `save_runtime_state()` (default=0). Cooldown sekarang dipersist ke Redis dan di-restore saat restart sehingga Q-Learning tidak langsung boost epsilon setelah restart di tengah episode degradasi.

---

### [xDS-5] Socket Leak di `is_reachable()`
**File:** `xds_server.py`

**Sebelum:** `sock.close()` dipanggil setelah `connect_ex()` — jika exception terjadi di antara keduanya, socket tidak pernah ditutup (leak).

**Sesudah:** Socket dijamin ditutup via `finally`:
```python
sock = None
try:
    sock = socket.socket(...)
    ...
    return sock.connect_ex((host, port)) == 0
except Exception:
    return False
finally:
    if sock is not None:
        sock.close()
```

Socket ini dipanggil hingga 3× per 3 detik (sekali per backend), sehingga leak ini relevan untuk server jangka panjang.

---

### [LP-4] `get_success_rate()` Dipanggil Tidak Perlu (Lazy Fix)
**File:** `qlearning/loop.py`

`get_success_rate()` sebelumnya dipanggil setiap cycle meski hanya dipakai saat `ep_sr is None` (fallback log). Sekarang dipanggil lazy — hanya saat endpoint SR tidak tersedia.

Dampak: **menghemat 2 Prometheus query per cycle** dalam kondisi normal (endpoint SR tersedia). Dengan `UPDATE_INTERVAL=7s` dan training berjam-jam, ini signifikan.

## 4. Catatan Desain: RT Cluster vs Per-Endpoint sebagai State

### Mengapa RT yang Dipakai adalah Cluster-Level

State ke-4 (`rt_level`) menggunakan **rata-rata RT cluster secara keseluruhan** dari Envoy, bukan RT per backend individu. Ini adalah keputusan desain yang disengaja.

#### Perbandingan opsi

| | Cluster RT (dipilih) | Per-endpoint RT (tidak dipilih) |
|-|----------------------|----------------------------------|
| Dimensi tambahan | 1 (rt_level: 0–3) | 3 (satu per backend) |
| State space | 5³ × 4 = **500** | 5³ × 4³ = **8.000** |
| Informasi | Latency keseluruhan ke user | Latency per backend |

State space 8.000 terlalu besar untuk training dalam batas waktu yang realistis.

#### Apakah cluster RT "tidak objektif"?

Bukan masalah objektivitas — ini soal **observabilitas**. RT cluster tetap valid sebagai state karena mencerminkan *dampak langsung action agent ke user*:

> Jika agent terlalu banyak mengirim traffic ke vm5 (backend lemah) → RT cluster naik → state berikutnya menunjukkan `rt_level=SLOW/CRIT` → agent belajar menghindari action itu dalam kondisi load yang sama.

Ini adalah feedback loop yang tepat untuk reinforcement learning.

#### Keterbatasan: ambiguitas kausalitas

State `rt_level=SLOW` tidak memberi tahu agent *penyebabnya*:

- **(A)** Backend tertentu memang lambat secara hardware, atau
- **(B)** Agent sendiri yang mengirim terlalu banyak traffic ke backend lemah

Namun keterbatasan ini *acceptable* karena:
1. Reward function sudah menangkap RT penalty secara langsung (bobot 0.50).
2. Dimensi CPU/RAM per-backend (dimensi 1–3) sudah memberi sinyal *mana* backend yang berat — sebagian mengkompensasi ambiguitas RT.
3. RT per endpoint pun tidak bebas dari efek yang sama: jika vm5 lambat karena agent kirim banyak traffic ke sana, RT vm5 ikut naik — tetap ada pengaruh policy agent, hanya lebih granular.

#### Kesimpulan

RT cluster adalah *consequence-aware signal* — agent mengamati akibat dari actionnya sendiri ke user, yang memang tujuan RL. Kombinasi dengan CPU/RAM per-backend cukup untuk memberi agent gambaran yang memadai tentang kondisi environment.

---

### Apa yang Terjadi dengan rt_level dalam Q-Learning

`rt_level` menjadi elemen ke-4 dari state tuple yang dikirim ke Q-table:

```
state = (level_vm3, level_vm4, level_vm5, rt_level)
        misalnya: (2, 1, 0, 2)  → MID / LIGHT / IDLE / SLOW
```

State ini dikonversi menjadi key Redis:

```
state_key = "2_1_0_2"
```

Karena state sudah menjadi 4-tuple, Q-table memiliki **entry terpisah untuk setiap kombinasi rt_level** meski beban CPU/RAM backend sama:

| State key | Kondisi | Q-values (vm3, vm4, vm5) |
|-----------|---------|--------------------------|
| `2_1_0_0` | MID/LIGHT/IDLE + RT **FAST** | belajar secara terpisah |
| `2_1_0_2` | MID/LIGHT/IDLE + RT **SLOW** | belajar secara terpisah |
| `2_1_0_3` | MID/LIGHT/IDLE + RT **CRIT** | belajar secara terpisah |

Artinya, agen belajar **kebijakan yang berbeda** untuk kondisi beban yang sama tapi dengan kualitas latency yang berbeda. Contoh:
- Saat load sama tapi RT=FAST → agen mungkin tetap di distribusi saat ini (exploit stabil)
- Saat load sama tapi RT=CRIT → agen belajar untuk segera pindah ke backend lebih ringan (eksplorasi agresif)

Tanpa dimensi RT, kedua kondisi itu dipetakan ke Q-table entry yang sama — agen tidak bisa membedakannya dan terpaksa menggunakan satu kebijakan untuk situasi yang seharusnya ditangani berbeda.

---

### Mengapa RT Tidak Dimasukkan ke Composite Score (W_CPU=0.4, W_RAM=0.6)

Alternatif yang mungkin dipertimbangkan adalah mengintegrasikan RT ke composite score:

```
# Composite saat ini — mengukur beban resource
score = W_CPU(0.4) × effective_cpu + W_RAM(0.6) × ram

# Alternatif — RT ikut dicampur
score = W_CPU × cpu + W_RAM × ram + W_RT × rt_normalized
```

Opsi ini **tidak dipilih** karena empat alasan:

**1. RT dan resource load adalah dua hal yang orthogonal.**
CPU/RAM mengukur *seberapa berat beban resource backend*. RT mengukur *kualitas layanan yang dirasakan user*. Keduanya tidak selalu berkorelasi:
- CPU rendah tapi RT tinggi → network latency, GC pause, atau request kompleks
- CPU tinggi tapi RT masih oke → request ringan tapi banyak

Mencampurnya dalam satu angka menghilangkan kemampuan agen membedakan kedua kondisi ini.

**2. RT jauh lebih volatile dari CPU/RAM.**
CPU dan RAM bergerak gradual. RT bisa spike tiba-tiba satu cycle lalu normal kembali, menyebabkan level backend loncat-loncat meski resource usage stabil. Volatilitas ini akan merusak kestabilan state.

**3. RT cluster tidak bisa masuk composite per-backend.**
Composite score dihitung per backend (vm3, vm4, vm5 masing-masing). RT yang tersedia adalah cluster-level (global dari Envoy) — tidak bisa dialokasikan ke satu backend tertentu. Untuk masuk composite, dibutuhkan per-endpoint RT dari Prometheus, yang menambah kompleksitas query.

**4. Informasi lebih kaya jika dipisah.**
Dengan RT sebagai dimensi terpisah, agen menerima sinyal yang tidak ambigu:

| Dimensi | Mengukur |
|---------|----------|
| vm3, vm4, vm5 | Beban resource per backend (dari CPU + RAM) |
| rt_level | Kualitas layanan yang dirasakan user (cluster RT) |

State space memang naik dari 125 ke 500, tapi agen mendapat gambaran environment yang lebih lengkap dan dapat diinterpretasi secara terpisah.
